## 1. 프로젝트 개요

미래에셋증권 사내 AI 챗봇 **Mi-Tone** 프로젝트. 회사 「고객중심 언어 가이드」(2026.02 발간)를 기반으로 직원이 작성한 메시지를 RAG + LLM으로 교정.

### 두 챗봇 시스템
1. **CX 챗봇 (Mi-Tone)**: 고객 메시지 교정 (SMS·LMS·알림톡·이메일·뉴스레터·VOC 답변)
   - production 프롬프트: `01_CX/PROD/system_prompt_v5.45_cx_flat_LF_escaped.txt` (57,918 byte)
   - vector DB: `01_CX/vector_db/`
     - `guide_rag_cx_docx_flat/` — 가이드 docx 44개 (cx_*.docx: 패턴/원칙/용어/채널)
     - `정답데이터2/` — 정답지 docx 273개 (cx_goldens2 LMS 249 + cx_voc_goldens VOC 4 + cx_exceptions 20)
     - `정답데이터1/` — 옛 1차 골든 7개 (cx_goldens_01a~01f.docx + xlsx, 사용자가 복구함)

2. **직원용 챗봇 (Mi-Tone 사내쪽지)**: 직원 간 쪽지 톤 보정
   - production 프롬프트: `02_직원용/PROD/system_prompt_internal_memo_v9.4.txt` (14,973 byte)
   - vector DB: `02_직원용/vector_db/internal_memo_guide_docx_v2/` — docx 22개 (CX 가이드에서 직원용 맞는 것만 추림)

3. **UX 시스템** (별도, 제 손길 닿지 않음)
   - vector DB: `03_UX/vector_db/guide_rag_ux_docx_flat/`

### 환경
- 모델: **GPT-OSS-120B** (한국어 처리 능력 양호 — 사용자 확인. 모델 한계 아님)
- RAG 한도: **65,258 byte 작동, 65,508 byte 실패** (시스템 프롬프트 byte 기준)
- 챗봇 프레임워크: 사용자가 명시 안 함 (Flowise/Weaviate **아님** — 사용자 정정)

### 모델 동작 메커니즘 (가장 중요)

**1. 메타 검증 룰 = 모델 부담 (역설적 결과)**
- "자가 검수 6항목"·"출력 전 3단계 검증" 같은 약 7,000 byte 메타 룰
- 모델은 시뮬레이션 안 함, 토큰만 차지 → 본문 출력 attention dilute → 누락 ↑
- v5.42에서 삭제 후 맞춤법·속도·에러 모두 개선 (사용자 확인)
- **교훈**: "검증해라" 룰 추가 신중. 효과 측정 안 되면 삭제 우선

**2. 출력 토큰 > 입력 토큰 (누락 영향)**
- 출력 형식 3줄 → 1줄 압축이 byte 축소보다 누락 차단에 효과 큼
- 출력 마지막 부분 attention dilute 위험

**3. 모델 자기 출력 인용 = 누락 전파**
- [교정 결과] 본문 출력 후 [수정 사항]에서 본문 그대로 복사 → 누락 전파
- v5.42에서 "본문 인용 금지" 룰로 차단

**4. EX 형식 = 출력 형식 (강한 anchor)**
- 정답 12선 `변환>` 형식이 출력 형식 명세보다 강한 anchor
- 안 일치시키면 EX 모방으로 출력 형식 회귀

**5. Anti-priming 위험**
- "❌ ~ 금지" 부정 예시가 오히려 모델이 그것을 생성 유도

**6. 정중 격상 ≠ 옛 어투**
- 모델이 "정중하게" 명령을 옛 표현(~하오니)으로 잘못 학습
- 명시적 차단 + 자연 합쇼체 anchor 필요

### RAG 활용

**7. RAG 사례 편향 = 출력 형식 편향**
- LMS 카드 194개 vs VOC 4개 → VOC 입력해도 LMS 우세 검색 → LMS 형식 모방
- "VOC 줄바꿈 안 됨" 진짜 원인 = RAG 편향 (단순 룰 부재 아님)

**8. 채널별 RAG 의존도 분리**
- 카드 풍부 채널: RAG 모방
- 카드 부족 채널: 프롬프트 인라인 룰 우선

**9. 도메인 분리 (CX vs 직원용)**
- CX: 고객 메시지, **PII 변환·■ 블록·5단 구조**
- 직원용: 직원 쪽지, **PII 보존·자연어·■ X**
- 같은 가이드 docx 사용해도 정반대 룰

### byte와 한도

**10. byte ≠ 출력 품질 (직접 비례 아님)**
- v5.41 -41% 축소했는데도 글자 누락 여전
- byte는 한도 초과(일시적 에러)에만 직접 영향
- 출력 품질은 출력 토큰 길이·메타 룰 부담이 핵심

**11. byte 한도 5,000+ 여유 확보 필수**
- v5.40 한도 2,348 byte 여유 → RAG 주입 시 잘림 → 일시적 에러
- v5.42 9,170 byte 여유 → 안정

### 작업 프로세스

**12. 시뮬레이션 ≠ 실제 모델 동작**
- 책상 reasoning trace로 33 케이스 검증해서 "PASS" 보고 → production에서 망가짐
- **사용자 환경 실제 출력만이 진짜 검증**

**13. 한 번에 한 가지 변경**
- v5.33 → v5.38 한꺼번에 5개 블록 변경 → 회귀 원인 격리 불가
- 단계적 1건 변경 → 사용자 환경 검증 → 다음

**14. 표면 검증 ≠ 진짜 검증**
- grep 매칭 PASS = 실제 동작 보장 X

**15. 사용자 추궁 후 새 문제 발견 패턴**
- 매번 사용자가 "제대로 했냐" 물을 때마다 새 문제 발견
- 사용자 사례를 책상 시뮬레이션보다 우선

### 사용자 의도

**16. 사용자 의도 확인 우선**
- 직원용 얘기인데 CX 만지작거린 사례
- "VOC 줄바꿈" 진단을 단순 룰 부재로 잘못 판단

**17. 톤 정책은 사용자 결정 우선**
- 직원용 v7 "톤 보존" → v8 "임원 기준 정중 격상" (사용자 정정)

### 기술적 룰

**18. LangChain 단일 중괄호 escape**
- `{변수명}` 단일 중괄호는 LangChain input variable로 해석 → 에러
- `{context}`, `{question}` 외 모든 단일 `{...}`는 `{{...}}` 이중 escape
- 직원용 v6에서 이걸 안 해 즉시 에러 발생

## 9. Cowork 작업 권고

### 우선순위
1. **사용자 환경 출력 사례 받기**
   - VOC 인사 누락 사례 1~2개
   - 옛 어투 회귀 사례 1~2개
   - 메타 룰 이름 출력 사례 1~2개
   - 출력 개행 안 됨 사례

2. **사례 기반 정확 진단**
   - 책상 시뮬레이션 X (인사이트 #12)
   - 사용자 환경 사례만이 진짜 데이터

3. **한 번에 한 가지 변경** (인사이트 #13)
   - 변경 → 사용자 환경 검증 → 다음
   - 멀티 변경 누적이 v5.34~v5.41 망친 원인

4. **byte 한도 여유 5,000+ 유지** (인사이트 #11)
   - v5.45 현재 7,340 byte 여유 (안전)

### 회피 패턴
- 메타 검증 룰 ("자가 검수해라", "3단계 확인해라") 추가 X
- 부정 예시 ("❌ ~ 금지") 나열 자제 (anti-priming)
- EX 형식과 출력 형식 일치 (정답 12선 `변환>` 형식 모방 위험)
- 직원용에 단일 중괄호 사용 X (반드시 `{{...}}` 이중 escape)

### 알려진 미해결 (v5.45 / v9.4)
1. CX VOC 첫 줄 인사 누락 가능성 (RAG LMS 편향)
2. CX·직원용 출력에 자의적 룰 이름 노출 가능성
3. 출력 형식 개행 보장 안 됨 (EX `변환>` 1줄 형식 anchor)
4. 직원용 옛 어투 회귀 (드물게)
5. 직원용 가끔 맞춤법 오류

### 시도하면 좋을 것
1. **사용자 환경에서 메시지 코드 명시 입력** (예: 카드 ID `[CH065]` 같이 첨부) → RAG 정확 매칭
2. **vector DB top_k 줄이기** (사용자 환경 측 작업)
3. **VOC 카드 추가** (4개 → 16~20개로 확대) — 가이드 67-77p, 123-126p 본문 기반
