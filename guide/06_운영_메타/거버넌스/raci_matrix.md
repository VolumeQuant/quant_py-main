> R = Responsible (실제 작업) / A = Accountable (최종 책임) / C = Consulted (자문) / I = Informed (통보 받음)
>
> ⚠️ **사용자 검증 필요**: 본 매트릭스의 부서명·담당자·연락처·게이트별 사인 권한은 사용자 회사 실제 조직도와 대조해 정정 후 사용. 본 문서는 골격 제안이며 실제 조직과 다를 수 있음.

## 🔑 핵심 메시지

### "AI가 잘못된 답을 줬을 때 누구 책임인가?" 답안

**최종 출력 채택은 현업 작성자**:
- Mi-Tone은 사전 가공 도구. 검수 통과 형태로 미리 가공해 주는 역할
- 현업이 결과를 검토하고 채택·수정·반려 결정
- 채택한 메시지가 잘못됐으면 1차 책임은 현업, 2차는 사후 분석으로 책임 영역 분담

**카드 콘텐츠 잘못이면**:
- 카드 작성자(HTS 개발팀)가 R, 금융소비자보호팀이 A
- 카드별 작성자·승인자·증거 링크가 메타에 박혀 있음 (cx_goldens2_*.docx의 메타데이터 섹션)

**LLM 환각 (G8 등)**:
- 모델 한계 (분산 책임)
- v5.22 후처리 검증기로 분리 트랙
- 단기적으론 카드 사례 학습으로 일부 완화

**RAG 검색 오작동**:
- IT 인프라팀(RAG 운영팀)이 R+A
- HTS 개발팀이 카드 docx 형식·이름은 일관되게 유지

**광고 규제·법령 위반**:
- 컴플라이언스가 A
- HTS 개발팀이 시스템 프롬프트·카드에 룰 박음
- 금융소비자보호팀이 사례별 검수

## 🚪 GO 게이트별 사인 책임

배포 전 통과해야 할 게이트와 사인 권한:

| 게이트 | 통과 기준 | 사인 권한 |
|---|---|---|
| 게이트 1: 사전 조건 | RAG 운영팀 3개 질문 답변 + PII 스캔 통과 | RAG 운영팀 + 정보보호부 |
| 게이트 2: 파일럿 메트릭 | 5건 파일럿 통과 | HTS 개발팀 |
| 게이트 3: 이해관계자 | 정보보호·컴플라이언스 검토 | 정보보호부장 + 컴플라이언스 임원 |
| 게이트 4: 본 평가 | 50건 × 15명 평가, p<0.05 | 금융소비자보호팀장 |
| 게이트 5: 카나리 | 10% → 30% → 50% 단계별 | HTS 개발팀 + 금융소비자보호팀 |
| 게이트 6: 전사 배포 | 100% 적용 | 임원 + 모든 이해관계자 |
