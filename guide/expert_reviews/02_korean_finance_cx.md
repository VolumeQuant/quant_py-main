## HIGH 칭찬 (잘된 점)

**[CX v6.0] 라인 269-273 — 마케팅 심의문구 3종 verbatim 보존**
> 「※ 투자 전 설명 청취 및 상품설명서, 약관 필독 / ※ 이 금융상품은 예금자보호법에 따라 보호되지 않습니다. / ※ 자산가격 변동, 환율 변동 등에 따른 원금손실(0~100%) 발생 가능 및 투자자 귀속」

## HIGH 비판 4건

### H1. CX v6.0 — VOC 답변에서 placeholder 환각 시 PII 대신 변수 노출 위험
**라인 154-158**:
> 「★ VOC (가이드 123p verbatim "고객 이름+고객님 호명 후 인사"): / 입력에 실제 호명 (김미래·홍길동 등) 있음 → "[실제 이름] 고객님, 안녕하세요" (원문 그대로, 변수화 X) / ... / 절대 금지: VOC 출력에 #{{고객명}} 등 placeholder 등장」

**문제**: VOC는 1:1 응대로 PII 변환 금지인데, **CX 채널 LMS·SMS·알림톡과 한 시스템에 통합**되어 RAG가 LMS 골든을 가져오면 모델이 `#{{고객명}}` 변수를 그대로 본문에 끌어다 쓸 위험이 매우 높음(인계 문서 #7 "VOC 4개 vs LMS 194개 RAG 편향" 명시). 실제 운영에서 VOC 답변에 `#{{고객명}} 고객님, 콜센터 응대 ...` 같은 답변이 고객에게 발송되면 **개인정보보호법 제15조(수집·이용 목적 외 사용)·금융소비자보호법 제17조(설명의무) 위반** + 명백한 응대 사고. 라인 158 "절대 금지"는 선언일 뿐, 실효성 있는 차단 게이트 부재.

**근거**: 개인정보보호법 제15조·제17조(목적 외 활용 금지), 금융소비자보호법 제17조(설명의무 — 1:1 답변에 변수 표기는 설명의무 위반), 금투협 표준영업행위준칙 제22조(민원응대 — 고객 신원 확인 후 응대).

**패치 제안**: 0단계 직후 **출력 직전 검증 게이트** 추가:
```
[VOC 출력 직전 의무 차단]
채널 = VOC 확정 → 출력 본문에 #{{ 또는 }} 문자열 발견 시
출력 폐기 + "VOC는 1:1 답변으로 변수 사용 불가. 입력 원문 호명을 그대로 사용해 주세요" 안내 의무.
부분 일치 검사: #{{고객명}} / #{{계좌번호}} / #{{URL}} / #{{YYYY.MM.DD}} 5종 + 일반 #{{ 패턴.
```

### H2. UX v6.0 — 자본시장법 제57조 단정 표현 enum 부족, "안전한"·"100%"·"검증된" 누락
**라인 157, 196-197**:
> 「광고 규제 위반어 — "수익률 N% 보장/가능/확정"·"원금 보장"·"무위험"·"확실한 수익"·"최고의"·"유일한"·"업계 1위"(객관 근거 없을 때)」

**문제**: 자본시장법 제57조 제2항 + 시행령 제60조·금투협 광고심의규정 제6조(금지표현)에서 명시한 단정·과장 표현은 위 9종 외에도 **"안전한"·"100%"·"손실 없는"·"보장된 수익"·"검증된"·"필수"·"반드시"·"무조건"·"누구나"·"한정 특가"** 등이 더 넓다. 특히 직원이 자주 쓰는 **"수익률 ○○% 가능"** 패턴 중 "보장" 없이 "가능"만 있는 경우(예: "연 7% 수익 가능합니다")도 제57조 위반(예측치 단정)인데 enum에 잡혀 있지 않음. 모델은 enum에 없으면 통과시킴.

**근거**: 자본시장법 제57조 제2항, 시행령 제60조, 금투협 광고심의규정 제6조 제1항·제2항·제3항, 금감원 「투자광고 가이드라인」 (2023.12 개정) p.14-17 단정·확정 표현 25종 enum.

**패치 제안**: 라인 157 위반어 목록 확장 + 정규식 패턴 추가:
```
- 단정·확정 표현 (\d+%|연\s?\d+%) + (보장|가능|확정|예상)
- 안전 표현: "안전한"·"안정적"·"손실 없는"·"무손실"·"리스크 없는"
- 강조 표현: "100%"·"무조건"·"반드시"·"필수"·"누구나"
- 비교 표현: "최고"·"최저"·"유일"·"업계 1위"·"독보적" (객관 근거 미첨부 시)
근거: 금감원 「투자광고 가이드라인」 (2023.12)
```

### H3. CX v6.0 — 「 」 외 괄호 금지 룰이 placeholder 대괄호와 충돌, 실효성 부족
**라인 393**:
> 「법령은 「 」 이중꺾쇠괄호만 (예: 「자본시장과 금융투자업에 관한 법률」). 《》, [], (), "" 등 다른 괄호 절대 금지. [참고자료] 정답지의 [X] 대괄호는 placeholder 표기일 뿐, 법령명에 적용 X」

**문제**: 같은 메시지 안에 **`[미래에셋증권]` 말머리·`[CTA 버튼]`·`(광고)` 표기·`(공백 포함)` 부기·`(토/일/공휴일 제외)` 영업시간** 등 정규 운영 양식이 모두 `()`·`[]` 사용. 모델 입장에서 "법령은 「 」만"·"다른 괄호 금지"가 충돌 시그널로 작용해 **법령명을 `()` 안에 넣거나 말머리 `[미래에셋증권]`을 「미래에셋증권」으로 변환하는 회귀**가 빈발. 인계 문서 미해결 #2와도 연결. 라인 393은 "법령명에 적용 X"라고 단서를 달았으나 모델 attention이 분산됨.

**근거**: 표준약관 작성지침(공정위 2022) 제3조 제4항 — 법령 인용 「 」, 약관·상품명 인용 ' ', 부기·예시 ( ). 금감원 「금융상품 약관 표준화 가이드」 제2조의3.

**패치 제안**: 룰을 "법령명만 「 」"이 아니라 **포지티브 enum**으로 재작성:
```
[괄호 사용 — 포지티브 enum]
- 법령명: 「자본시장과 금융투자업에 관한 법률」 (이중꺾쇠 「 」 only)
- 회사명 말머리: [미래에셋증권] (대괄호)
- 광고 표시: (광고) (소괄호)
- 영업시간 부기: (평일 08:00~17:00) (소괄호)
- CTA 버튼 표시: [버튼명] (대괄호)
- 약관·상품명 인용: '연금저축계좌' (작은따옴표)
이 외 자발 변환 금지. 입력 원문 괄호 그대로 보존.
```

### H4. 직원용 v11.0 vs CX v6.0 — PII 정반대 원리 충돌, 입력이 잘못 들어가면 사고
**직원용 라인 169**:
> 「PII 원문 그대로 보존 (직원 쪽지 도메인 — 변수화 절대 X): 김*영, 010-XX**-**XX, 계좌번호, URL, 날짜」

**CX v6.0 라인 122-130**:
> 「변수 변환 — LMS·SMS·알림톡·이메일·뉴스레터 채널 한정 / 마스킹 호명 (김*영·박*수) → #{{고객명}}」

**문제**: **같은 회사·같은 가이드를 기반으로 한 두 시스템이 PII 처리에서 정반대 원리** — 직원용은 보존, CX는 변환. 실무에서 **직원이 VOC 답변 초안을 직원용 챗봇에 잘못 넣으면 PII가 그대로 LMS 발송되거나, LMS 초안을 직원용에 넣으면 PII가 변수화 안 됨**. 두 시스템 어디에도 "이 메시지는 다른 시스템 대상" 감지 게이트 없음. CX v6.0 라인 15 "UX 시스템 안내"는 있으나 **CX↔직원용 상호 라우팅 안내 부재**. 개인정보보호법 제29조(안전조치 의무) 관점에서 시스템 경계 누설 위험.

**근거**: 개인정보보호법 제29조(안전성 확보 조치), 금감원 「개인정보 처리 방침 가이드」, 신용정보법 제19조(기술적·관리적 보호조치).

**패치 제안**: 두 프롬프트 모두에 **상호 라우팅 게이트** 추가:
```
[CX v6.0]
입력에 "사내 쪽지"·"부장님"·"팀장님"·"슬랙"·"AP변경"·"TR-" 코드 패턴 발견 시:
"직원 간 사내 메시지로 보입니다. 직원용 Mi-Tone 시스템을 이용해 주세요."
[직원용 v11.0]
입력에 "[미래에셋증권]"·"#{{고객명}}"·"(광고)"·"고객님" 첫 줄 패턴 발견 시:
"고객 대상 메시지로 보입니다. CX Mi-Tone 시스템을 이용해 주세요."
근거: 개인정보보호법 제29조 안전조치 의무
```

## MEDIUM 비판 3건

### M1. CX v6.0 — VOC 정정 공시·면책 문구 누락 가능성
**라인 84-100 (VOC 채택 시 의무) + 라인 671-673 (EX VOC 답변)**:
> 「내용 구성: 고객 감정/상황에 공감 → 조치방안 안내 → 재발 방지 의지 표명 → 클로징 순서」

**문제**: VOC가 **사실 정정·정정 공시 사항을 담아야 할 때(이전 안내가 잘못된 경우)** 4단 구조에 "정정 사실 명시·이전 안내 무효화·면책 사유 고지" 단계가 없음. 라인 97 "사실 정정 → VOC_misclaim (가이드 125p)" 구분이 있으나 정정 공시 의무 표현(예: "○○일자 안내드린 내용 중 ○○ 항목을 다음과 같이 정정합니다") verbatim 템플릿 부재. 금융소비자보호법 제17조(설명의무) + 제19조(부당권유 금지) 정정 공시 표준 표현 누락 시 민원 재발 + 금감원 분쟁조정 시 회사 패소 위험.

**근거**: 금융소비자보호법 제17조 설명의무, 제19조 부당권유 금지, 금감원 「금융소비자 보호 모범규준」 제22조 정정 공시 의무.

**패치 제안**: VOC 하위 유형에 `VOC_correction` 추가:
```
- 사실 정정 + 정정 공시 → VOC_correction (가이드 125p, 금소법 제17조)
  필수 4단:
  ① "○○년 ○월 ○일 안내드린 내용 중 ○○ 항목을 다음과 같이 정정합니다"
  ② 정정 전·후 명시
  ③ 정정 사유
  ④ 정정 안내 누락 책임 사과 (1회) + 향후 재발 방지
```

### M2. 직원용 v11.0 — "당사 → 미래에셋증권" 룰이 직원 간 쪽지에 과도, 회사명 prefix 부담
**라인 142**:
> 「당사 → 미래에셋증권 (또는 생략)」

**문제**: 직원 간 사내 쪽지에서 "당사" 사용은 외부 자료 인용 시 자연스러움(예: "당사 약관에 따르면"). 이를 무조건 "미래에셋증권"으로 풀어쓰면 **사내 쪽지가 외부 안내문 톤이 되어 부자연스럽고**, 직원이 챗봇 신뢰 저하. CX 도메인(고객 대상)은 prefix 의무가 맞으나, 직원용은 가이드 134p 한자어 순화 본 의도(고객 이해 돕기)와 **목적 충돌**. 가이드 32p (직원용 적용 룰 enum) 본문에 "당사 → 미래에셋증권" 제외 검토 필요.

**근거**: 가이드 134p 한자어 순화 — 고객 이해 목적, 사내 쪽지는 "당사" 자연 통용. 가이드 9p Trustworthy "차분한 정보 전달" — 사내에서는 "당사"가 더 차분.

**패치 제안**:
```
[직원용 — 한자어 순화 도메인 조정]
- "당사" → 사내 쪽지에서는 보존 (외부 인용 시 자연스러움)
- 단, "당사는 ~합니다" 형태로 회사 자체 주어인 경우만 "미래에셋증권"으로 풀어쓰기
- 외부 자료 인용 ("당사 약관·당사 규정") → 보존
```

### M3. UX v6.0 — `[참고자료]` placeholder `[X]`와 법령명 「 」 룰 충돌 명시 부족
**UX v6.0 라인 164**:
> 「⚠️ 법령명: 「 」 이중꺾쇠괄호만. 《》/[]/() 등 다른 괄호 절대 금지. [참고자료] 정답지의 [X] 대괄호는 placeholder 표기일 뿐, 법령명에 적용 X」

**문제**: UX 시스템은 `[종목명]·[회사명]·[고객명]` placeholder를 권장(라인 285) 사용하면서, 동시에 `[]` 괄호 금지를 명시. **모델이 보기에 `[종목명]`은 OK, `[자본시장법]`은 NG** 인 차이를 attention이 잡기 어려움. 인계 문서 미해결 #2 (자의적 룰 이름 출력)와 동일 카테고리 — 룰 우선순위 명시 부족. UX는 글자수 제약 1글자 탈자도 치명적인 도메인이라 이 충돌이 즉시 표면화.

**근거**: 가이드 79p 띄어쓰기·기호 규칙, 표준약관 작성지침 제3조.

**패치 제안**:
```
[괄호 우선순위 — UX 도메인]
1. 법령 인용: 「자본시장법 제57조」 (이중꺾쇠 — 한 메시지 1회 이상 등장 시 의무)
2. UI placeholder: [종목명]·[회사명]·[고객명] (대괄호 — 가이드 예시 인용용)
3. 부기·괄호: () (가이드 표기 그대로)
충돌 시 1번 우선. 법령명을 [ ]·( )로 변환 절대 금지.
```

### L1. CX v6.0 — ☎ 기호 처리 verbatim 룰이 모델 환각 유발 가능
**라인 178-181**:
> 「☎ 기호는 모든 채널에서 완전 삭제 (가이드 79p 기호 룰 부합) / LMS·SMS·알림톡·이메일·뉴스레터: ☎ 삭제 후 ■ 문의 블록 안에 "미래에셋증권 [센터명] [번호]" 형태로 정리」

**문제**: 라인 196 "원문에 전화번호 없으면 ■ 문의 블록 자체 생성 금지 (환각). 임의의 1588 번호 추가 절대 금지" 룰이 있으나, 직전 라인 187 "신호 → 블록 / ☎ 전화번호 발견 → ■ 문의" 매칭 룰이 있어 **모델이 ☎ 기호 자체를 트리거로 받아 번호가 마스킹·placeholder여도 1588-6800·1644-9300 같은 실제 번호 환각** 위험. 가이드 79p verbatim보다 **구체 번호 안전 표 추가**가 필요.

**근거**: 가이드 79p 기호 규칙, 인계 문서 운영 룰 [6] "☎ 회사명 prefix 누락 환각 차단".

**패치 제안**:
```
[☎ 처리 — 환각 방지 강화]
- 원문 ☎ 다음에 구체 숫자 X → ■ 문의 블록 생성 금지 (가이드 79p)
- 원문 ☎ + 마스킹 번호 (010-XX**-**XX) → ☎ 삭제, 번호는 #{{고객번호}} 변환 (별도 ■ 문의 X)
- 원문 ☎ + 회사 대표번호 (1588-6800·1644-9300·1588-5577) → ■ 문의 + "미래에셋증권 [센터명] [번호]"
- 임의 추가 금지 enum: 1588-6800 / 1644-9300 / 1588-5577 / 1644-6446
```
