### HIGH 1. [CX 388-399행] CTA 동사 라벨 길이 제약 룰이 누락됨
**라인 인용** (CX 388-399, "절대 보호" 8종):
> `1. 숫자·사실(상품명·금액·수익률·날짜·계좌·수량) 원문 그대로`
> ... (글자수 한계 0개)

**원칙 근거**: 카카오 알림톡 채널은 가이드 282행에서 "CTA 버튼: 공백 포함 14자 이내" 1줄만 명시. SMS 45자, LMS 1,300자는 본문 한계로 분리되어 있으나 **버튼 라벨**의 글자수·동사 라벨 룰은 채널별로 흩어져 있어 모델이 검색하기 어렵습니다. UX 라이팅에서 CTA는 동사·간결·결과 지향(verb-first, scannable)이 핵심 — 토스 디자인 시스템도 CTA를 별도 토큰으로 관리합니다.

**구체 패치**: CX 프롬프트 392-399행 "절대 보호 8종" 다음에 "글자수 절대 한계 표(체크리스트)" 박스 신설:
```
[글자수 절대 한계 — 채널×컴포넌트]
- SMS 본문: 45자 (말머리 포함, 가이드 59p)
- 알림톡 CTA 버튼: 14자 (공백 포함, 가이드 65p)
- LMS 제목 라인: ~30자 권장 (가이드 60p)
- VOC 답변 전체: 400자 (가이드 123p)
```

```
- 바텀시트 타이틀     20자 이내
- 바텀시트 본문 항목  한 줄 25~30자
```

### HIGH 3. [UX 94-96행] 헬퍼 텍스트 vs 에러 메시지 분기는 좋으나 "성공 피드백" 톤이 빠짐
**라인 인용** (UX 94-96):
> `[헬퍼 텍스트 이중 톤 — 가이드 87p]`
> `- 입력 안내 (정확한 입력 돕는 힌트) → 해요체 + ~해 주세요`
> `- 유효성 검증 실패 (피드백) → 하십시오체 (보안과 직결될수록 객관적 태도로 시스템 신뢰감)`

**원칙 근거**: UX 마이크로카피의 4가지 상태(빈/로딩/에러/성공) 중 **성공 피드백**(예: "비밀번호가 정상 설정되었습니다") 톤이 명시 안 됨. 토스 가이드는 성공 피드백을 "짧은 명사형 + 마침표 생략"으로 표준화. 현재 프롬프트는 99-101행 "마침표 규칙"에 단편적으로 흩어져 있어 모델이 분기를 통합 인식하기 어렵습니다.

**구체 패치**: UX 94-96행을 "헬퍼 텍스트 3분기"로 확장:
```
[헬퍼 텍스트 3분기 — 가이드 87p]
- 입력 안내 (default): 해요체 + ~해 주세요 + 마침표 부여
- 검증 성공 (positive): 명사형/하십시오체 + 마침표 생략 (예: "사용 가능한 비밀번호")
- 검증 실패 (error): 하십시오체 + 원인 명확 + 마침표 부여 (가이드 100p Recovery)
```

### MEDIUM 1. [CX 60-76행] "유형 확인 부탁드립니다" 다이얼로그 본문이 7줄로 모바일 한계 초과
**라인 인용** (CX 60-76):
> `## [유형 확인 부탁드립니다]`
> ... `1. **SMS** — 45자 이내 단문 안내`
> ... 7개 옵션 + 안내 + 참고문 (총 약 17줄)

**원칙 근거**: 챗봇 답변이지만 모바일 채팅 UI에서 한 말풍선 17줄은 스크롤 부담. 정보 위계(Headline > Body > Caption) 관점에서 핵심 질문 1줄 + 옵션 7개 + caption 1줄로 압축 가능. 또한 옵션 설명에 "1,300자 이내"(67행) 같은 메타 수치가 노출되어 직원이 헷갈릴 수 있음 — UX 라이팅 원칙 "사용자가 결정에 필요한 정보만"(progressive disclosure) 위반.

**구체 패치**: 옵션 설명을 동작 중심 1줄로 단축:
```
1. **SMS** — 45자 이내 짧은 안내
2. **LMS 공지** — 항목 구조의 일반 공지
3. **LMS 마케팅** — (광고) 표기 캠페인
... (각 줄 20자 이내)
```

### MEDIUM 3. [직원용 175-184행] "하지 않는 것" 박스가 부정문 9개 나열 — anti-priming 위험
**라인 인용** (직원용 175-184):
> `- 입력에 없는 인사·마무리 자발 추가 X (가이드 미명시)`
> `- 입력에 없는 호칭 자발 추가 X`
> `- "고객님" 자발 추가 X (직원 쪽지 도메인, CX 36p는 적용 X)`
> `- ■ / ※ / * 블록 구조로 재포맷 X (CX 형식)`
> ... (총 9개 X 항목)

**원칙 근거**: HANDOFF 인사이트 #5 "Anti-priming 위험 — 부정 예시가 오히려 모델이 그것을 생성 유도"가 명시. UX 라이팅에서도 negative phrasing보다 positive guidance가 효과적("Don't do X"보다 "Do Y"). 9개 부정 항목 중 "■ 블록 X" 등은 CX 형식 재유입 트리거가 될 수 있음.

**구체 패치**: 부정 형식을 긍정 룰로 재작성:
```
[보존 우선 룰 — 직원용 도메인]
- 입력 텍스트 그대로 다듬기 (인사·호칭·마무리 자발 추가 없음)
- 자유 텍스트 형식 유지 (■/※ 블록 사용 안 함, 항목 3개 이상은 - 리스트)
- 직원 간 호칭 보존 (부장님·과장님 등 입력 그대로)
```

### LOW 1. [CX 전반] 약어 풀어쓰기·접근성 가이드 부재
**라인 인용** (CX 316행, 직원용 165-166행):
> `사내 업무 용어(ISA·IRP·ELS·ETF·CFD·RP형·DC·DB형·TR-XXXX) 원어 유지`

**원칙 근거**: 사내 약어 보존은 맞으나, **고객용 메시지(SMS·LMS·이메일)** 에서는 첫 등장 시 풀어쓰기 가이드(WCAG 3.1.4 Abbreviations 정신)가 빠짐. 미래에셋 고객 중 시니어·초보 투자자도 있고 스크린 리더 사용자에게 "ISA"는 "아이에스에이"로 음독되어 의미 전달 실패. 가이드 27p "전문용어 쉽게 표현"과도 직결.

**구체 패치**: CX "한자어 순화 — 가이드 134p" 박스 다음에 추가:
```
[금융 약어 — 첫 등장 풀어쓰기 권장 (가이드 27p)]
고객 채널(SMS 제외) 첫 노출: "개인종합자산관리계좌(ISA)" 형태
사내 채널(직원용) 및 SMS 글자수 한계: 약어 그대로
```

### LOW 2. [UX 156행] 한자/특수문자 절대 보호 룰에서 색 대비·터치 타깃 가이드 누락
**라인 인용** (UX 156-162):
> `1. 브랜드 표기 — M-STOCK(대문자, 하이픈)·미래에셋증권·카이로스(KAIROS)`

**원칙 근거**: UX 라이팅 시스템이지만 카피만 다룰 뿐 "**카피와 디자인 토큰의 접점**"(예: 부정 강조 시 빨강만 의존 X, 색맹 사용자 대비 텍스트 prefix "주의:" 권장)이 빠짐. 본 프롬프트가 다루는 범위가 카피 한정이라면 명시적 비범위 안내가 좋고, 그렇지 않다면 접근성 한 줄 권장이 필요합니다.

**구체 패치**: UX 153-164행 "절대 보호" 끝에 1줄 추가:
```
8. 접근성 — 부정/오류 카피는 색에만 의존하지 말고 텍스트 prefix("오류:", "주의:")로 의미 전달 (WCAG 1.4.1 정신, 가이드 100p Recovery)
```
