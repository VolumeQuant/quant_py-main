# 시스템 프롬프트/Few-shot 엔지니어링 전문가 리뷰

대상: CX v6.0 (1,079 lines) / 직원용 v11.0 (470 lines) / UX v6.0 (626 lines)
모델 가정: GPT-OSS-120B (instruction-following 약·한국어 OK)

---

## 칭찬 (1건)

- **CX 라인 407~417 "RAG 4계층 분리"는 본 시스템 전체에서 가장 견고한 설계**. `① 구조 / ② 표현 / ③ 어휘 / ④ 사실` 분리 + "사실 토큰은 입력에서만"이라는 anchor는 GPT-OSS-120B 같은 약-IF 모델이 RAG 카드 사실값을 그대로 베끼는 환각을 차단하는 사실상 유일한 구조적 룰. 이 한 블록만으로 "카드는 형식 anchor일 뿐 사실 anchor가 아님"이 prepend로 박혀, 후속 [LMS Golden Verbatim 재현 룰] anchor와 균형이 맞음. 다른 두 파일에는 이 등급의 분리 룰이 없음(직원용에는 RAG 룰 자체가 거의 없고, UX 134~150 치트시트는 "사실 토큰" 개념이 없음).

---

## 비판 Top 9 (high → low)

### [HIGH-1] EX 형식과 [수정 사항] 형식 정의 충돌 — anchor 분열

- **위치**: CX `system_prompt_v6.0_cx_flat_LF_escaped.txt` L589-590 vs L944-948 / 직원용 `system_prompt_internal_memo_v11.0.txt` L285-286 vs L361-364 / UX `system_prompt_v6.0_ux_flat_LF_escaped.txt` L327-328 vs L531-532
- **문제**: 형식 명세 "원문" → **"교정"**" (별표 볼드 강제)인데 EX 본문은 별표 없는 평문. CX L944 `1. "당사는 관계규정에 따라" → "미래에셋증권에서는"  ·  ...` 직원용 L361 `1. "익일" → "다음 날"  ·  ...` UX L531 `1. "네 / 아니요" → "취소 / 확인"  ·  ...` 모두 형식 명세를 위반. 직원용·UX는 ★볼드 강제★ 라인을 두 군데(L298, L373)에 명시.
- **위험**: 인사이트 #4 "EX 형식 = 출력 형식 (강한 anchor)" — 약-IF 모델은 EX 형식이 명세보다 우선. 결과: 사용자 환경에서 볼드(**) 누락 회귀 거의 확정. 이게 인계 문서 #3 "출력 개행이 보장되지 않을 가능성"의 메커니즘과 동일.
- **개선안**: EX 12선·7선 전부에서 `**"..."**` 별표 추가 sweep. 형식 룰을 텍스트로만 두지 말고 anchor에 박을 것.

### [HIGH-2] Anti-priming 위험 — "절대 금지" 메타 룰 이름 나열이 그 표현 생성 유발

- **위치**: CX L843-844 / 직원용 L215-217 / UX L187-189
- **문제**: 세 파일 모두 "치트시트 / 자가 검수 / 검증 단계 / 자동 변환 / 암묵적 교정 N항 / 규칙 N번" 같은 금지 표현을 verbatim 나열. CX L844 `치트시트 / 자동 블록 생성 규칙 A/B/C / PII 변환규칙 / URL 변환규칙 / 암묵적 교정 N항 / 규칙 N번 / 자가 검수 / 검증 단계 — 모두 출력 금지`.
- **위험**: 인사이트 #5 + 인계 문서 회피 패턴 명시. 약-IF 모델은 "금지" 명령보다 token 자체를 먼저 학습 → 사유 칸에서 그대로 생성. 더불어 직원용 L201 메타 문구 6종 `"위 N번 포함" / "형식 유지" / "(공백 유지)"` 도 동일 위험.
- **개선안**: 부정 enum 삭제 → 긍정 화이트리스트로 전환. "사유 칸은 다음 어휘만 사용: 한자어 순화 / 정중 어미 자연화 / 번역투 제거 / 띄어쓰기 / 시스템 주체 피동 / 포용 언어." 토큰 등장 횟수도 1회로 제한.

### [HIGH-3] Negation 명령 중첩 — 약-IF 모델 무시 빈번

- **위치**: CX L78-82, L91-94, L162, L196-197, L212, L501-507, L513, L843, L901-905 / 직원용 L173-184, L199-202, L209-217, L259, L295-308 / UX L271-275, L380-389, L462-466
- **문제**: "절대 금지·X·금지·환각 판정·즉시 재작성·N건 재계산" 형 부정 명령이 한 파일 30건+. 예: 직원용 L295-308 한 박스 안에서만 "절대 금지" 4회. CX L204 `"■ 주요 내용 / ■ 진단 내용 / ■ 확인 및 응답 / ■ 유의사항 / ■ 내용 — 환각 판정"` 직원용 L180 `"■ 블록 사용 절대 금지"`.
- **위험**: instruction-following 약한 모델은 부정 명령을 일관 무시(인사이트 함의). "환각 판정"은 모델이 시뮬레이션 안 함 — 메타 검증 룰 잔재(인사이트 #1)와 같은 패턴.
- **개선안**: "X 금지" → "Y만 사용" 으로 모두 변환. ex) L515-517 "■ 주요 내용·■ 진단 내용 ... 자발 추가 절대 금지" → "허용 ■ 명칭은 다음 30종 중 하나(L204-211 enumeration)".

### [HIGH-4] 메타 검증 룰 잔재 — 토큰 낭비 + attention dilute

- **위치**: CX L488-489 `"한 샘플 한 번 탈자가 발견되면 4번 반복 고착 위험. 출력 직전 본문 명사 1단어씩 원문과 대조 권장"`, UX L182-185 `"출력 직전 본문 단어 단위 원문 대조 권장"`
- **문제**: 인사이트 #1에서 v5.42 시 "자가 검수 6항목·출력 전 3단계 검증 7,000 byte 삭제 → 개선 확인" 됐는데, **CX L488(2번 룰)·UX L182(3번 룰)에 같은 패턴이 부활**. "출력 직전 X와 대조" 는 모델이 시뮬레이션 안 하는 전형적 메타.
- **위험**: 출력 attention dilute, 누락 ↑.
- **개선안**: 대조 룰 두 줄 모두 삭제. 대신 [4] 가짜 원문 차단 룰 한 줄로 압축.

### [MEDIUM-1] {{}} brace escape — 운영 placeholder 호환 룰의 자기모순

- **위치**: CX L164-165
- **문제**: `"회사 운영 시스템은 단일중괄호 (샵 다음 중괄호 고객명 중괄호) 사용. 본 프롬프트는 LangChain 호환 위해 이중중괄호 표기지만, 운영 입력에 단일중괄호 들어오면 동일 placeholder로 인식해 한 글자도 변경 없이 보존"`
- **위험**: LangChain 단계에서 `#{고객명}` 단일중괄호가 input variable로 해석돼 KeyError 발생(인사이트 #18). 룰은 "둘 다 인식" 인데 LangChain 자체가 단일을 먹어버려 모델이 본 시점에서 이미 망가져 있음. 모델 룰로 해결 안 되는 영역에 모델 룰을 두는 self-contradiction.
- **개선안**: LangChain pre-processor 단계에서 `#{...}` → `#{{...}}` substitution을 먼저 적용. 시스템 프롬프트 레벨 룰은 삭제(또는 "이미 normalize 됐다고 가정" 1줄로 축소).

### [MEDIUM-2] 출력 형식 다중 정의 — 한 형식·여러 정의

- **위치**: CX L582-590 (케이스 1 골격) / L594-622 (형식 예 1 — 별표 볼드 있음) / L944-948 (EX-T1 — 별표 볼드 없음) / L815-834 (★형식 룰★) / L836-841 (■ 꼭 확인해 주세요 패턴) / 직원용 L283-286 / L290-308 / L312-329 / L361-364
- **문제**: CX는 동일 출력 형식을 5번 재정의. 형식 예 1~8(L594-799)·EX-T1 4건(L920-1032) 모두 미세하게 다른 anchor를 박음. L612 `(가이드 134p)`, L944 `(가이드 9p, 134p)` 처럼 같은 한자어 순화에 다른 페이지 인용도 발생.
- **위험**: token budget 낭비(약 4-5K byte 중복) + 모델은 가장 마지막·가장 빈번한 형식을 모방 → EX 12선이 anchor 승리 → HIGH-1 결과로 이어짐.
- **개선안**: 형식 골격 1건만 prepend, 형식 예 8개는 채널별 1개씩만(LMS·VOC·SMS·알림톡 4개). EX-T1 4건은 형식 예와 통합. 약 3-4K byte 절감.

### [MEDIUM-3] Multi-turn carry-over 차단 룰 — 단발성 명령 한계

- **위치**: CX L23-24
- **문제**: `"현재 입력에 등장하지 않는 placeholder·전화번호·계좌번호·회사명·서비스명·도메인은 출력 절대 금지. 같은 세션 이전 turn 토큰이라도 현재 입력에 없으면 환각으로 판정. 매 입력은 독립 처리."` — 직원용·UX에는 동일 룰 부재. 룰은 prepend 1회로만 등장.
- **위험**: GPT-OSS-120B는 multi-turn KV cache로 이전 turn placeholder를 자연스럽게 회수. "환각 판정" 메타 명령은 시뮬레이션 안 함. 직원용·UX에서는 multi-turn 보호 자체 없음.
- **개선안**: 시스템 측 chat template에서 turn 별 system message 재주입(implementation 레벨). 그래도 프롬프트에 두려면 직원용·UX에도 동일 룰 복제, 그리고 `"매 입력은 독립 처리"` 1줄로만 압축.

### [LOW-1] Token budget vs 룰 중복 — 같은 룰 3회+ 반복

- **위치**:
  - CX 한자어 순화: L309-316 + L620-622 + L944
  - CX 띄어쓰기: L335-347 + L781 + L808-811
  - 직원용 한자어 순화: L139-146 + L380-384
  - "■ 꼭 확인해 주세요 verbatim": CX L198, L836-841 / 직원용 L330 / UX L385-389
- **문제**: 같은 룰이 룰 박스 + 형식 예 + EX 본문 3회 등장. byte 누적은 인계 문서 5K 여유 한도 압박.
- **위험**: 룰 박스 vs EX 본문 wording이 미세 차이 발생 시 모델이 어느 쪽을 따를지 분열. 인사이트 #11 "byte 한도 5,000+ 여유 확보"와 직접 충돌.
- **개선안**: 룰 박스 = 정의 / 형식 예 = 1건 / EX = 사례 — 역할 분리. 중복 wording 제거.

### [LOW-2] RAG 4계층 분리 룰의 일관성 — 직원용·UX에 누락

- **위치**: CX L407-417 (있음) / 직원용 (없음) / UX (없음)
- **문제**: CX의 `4계층 분리(① 구조 ② 표현 ③ 어휘 ④ 사실)`가 다른 두 파일에는 부재. 직원용은 RAG 카드 자체가 적어 영향 작지만, UX는 가이드 docx + UX 정답 카드 RAG 사용함에도 사실/패턴 분리 룰 없음.
- **위험**: UX 카드 사실 토큰(글자수·페이지번호) 환각 가능. 직원용도 22 docx 카드의 PII가 RAG로 끌려와 출력 노출 위험.
- **개선안**: 4계층 분리 룰 핵심 5줄을 직원용·UX에도 prepend.

### [LOW-3] 시스템·고객 주체 피동 룰 — 능동/피동 ambiguity

- **위치**: CX L319-329 / 직원용 L133-136 / UX L62-68
- **문제**: 룰은 "시스템 주체 = 피동, 고객 주체 = 능동". 그러나 CX L778 사례 `"미수금 안내드립니다" → "미수금이 발생되었습니다"  ·  시스템 주체 피동 (가이드 99p)` — "안내" 행위 주체는 회사(능동)인데 사례는 "발생되었습니다" 결과 피동으로 변환. 룰과 사례 의미 mismatch. 직원용 L134 verbatim 인용 + L136 `"가입했습니다·신청하셨습니다"` 도 같은 ambiguity.
- **위험**: 모델은 사례 우선 학습 → 회사 능동 행위도 피동으로 변환할 위험.
- **개선안**: 사례에 의미 layer 명시 ("미수금이 = 시스템이 자동 인식한 상태값" 같은 1줄 주석). 또는 사례 자체 교체.

## 종합 평가

세 프롬프트는 가이드 본문 충실도와 도메인 룰 정밀도는 우수하지만, **prompt engineering 관점에서는 anchor 분열(HIGH-1)·anti-priming(HIGH-2)·negation 중첩(HIGH-3)의 3가지 구조적 결함이 누적**돼 약-IF 모델인 GPT-OSS-120B가 일관된 출력을 내기 어려운 상태. 인계 문서가 인사이트로 정확히 진단한 패턴(#1 메타 검증, #4 EX anchor, #5 anti-priming)이 본 v6.0/v11.0 텍스트에서 부분적으로만 적용됐고, 특히 EX 본문에 볼드(**) anchor 누락이 결정적이라 사용자 환경 회귀의 핵심 메커니즘으로 보임. 우선 HIGH-1 sweep(EX에 별표 추가) → HIGH-3 부정→긍정 변환 → MEDIUM-2 형식 통합 순서로 1회 1건 변경 + 사용자 환경 검증을 권장. byte 절감 약 4-6K 가능하므로 RAG 한도 여유도 동시 확보.
