## 비판 9개

### [HIGH-1] LMS 마케팅 심의문구 자동 "자발 추가" — 한국금융투자협회 광고심의 미경유 광고 송출 위험
- CX L267~273, L640~645, L653 verbatim:
  `(심의문구 3종 없음) → ※ 투자 전 설명 청취 / ※ 예금자보호 X / ※ 원금손실 가능 · 심의문구 3종 자발 추가 (가이드 63-64p, 자본시장법 제57조)`
- 자본시장법 제57조 제2항 및 금투협 「금융투자회사의 영업 및 업무에 관한 규정」 제2-26조(광고 사전심의)에 따라, 모든 투자광고는 협회의 사전심의를 거쳐야 한다. **LLM이 심의문구를 "자발 추가"하면 해당 광고는 협회 심의안과 다른 본문이 되어 미심의 광고로 송출**되며, 심의위반 과태료(자본시장법 제449조 제1항 제29호) 대상이다. 또한 예금자보호법 제2조(보호 여부 표시) 문구는 상품별로 정확한 문구 의무가 있다(예: 일부 ELS는 "보호되지 않습니다"가 잘못된 표기일 수 있음).
- **패치**: "자발 추가" 룰 삭제. 심의문구 누락 시에는 ⚠️ 경고만 출력하고 본문은 변경하지 않으며, "협회 광고심의 통과본 그대로 붙여넣어 주세요" 안내. 케이스 2(광고 규제 경고)와 통합하여 `심의문구 누락 → 자동 추가 X / 운영자 입력 의무` 로 변경.

### [HIGH-2] PII 원문 보존 정책의 개인정보보호법 제15조·제17조·제22조, 신용정보법 제32조·제33조 충돌
- 직원용 L165~169 verbatim:
  `PII 원문 그대로 보존 (직원 쪽지 도메인 — 변수화 절대 X): 김*영, 010-XX**-**XX, 계좌번호, URL, 날짜`
- CX L386~403 EX3 verbatim에서 LLM이 `김*영 고객 계좌 110-12-345678` 원문 출력.
- HANDOFF L160 `PII 변환 절대 금지 (변수화 X, 원문 보존)`.
- 개인정보보호법 제15조 제1항(수집·이용 적법근거), 제17조(제3자 제공 동의), 제22조(고지·동의), 제28조의2(가명처리)와 신용정보법 제32조(신용정보 제공·이용 동의) 위반 위험. **사내 쪽지가 외부 LLM(GPT-OSS-120B)으로 전송되는 순간 제17조의 "제3자 제공"에 해당**하며, 계좌번호·고객명은 신용정보법 제2조(개인신용정보)다. 외부 위탁 LLM의 경우 전자금융감독규정 제13조(외부주문 등 정보처리시스템 안전성 확보)·제14조의2(클라우드컴퓨팅서비스) 적용으로 보호조치 의무가 발생한다.
- **패치**: 직원용 PII 정책을 "쪽지 도메인이라도 LLM 입력 전 고객 식별 PII는 자동 마스킹 또는 거부" 로 변경. 입력 sanitizer를 시스템 프롬프트 외부(파이프라인)에 두고, 프롬프트 안에는 `계좌번호/주민번호/휴대폰번호/실명 패턴 발견 시 → 거절 + 마스킹 후 재입력 요청` 룰 추가. EX3는 마스킹 후 출력 사례로 교체.

### [HIGH-3] 면책·투자유의 문구 부재 — 금소법 제17조~제19조, 자본시장법 제49조, 제178조
- UX L391~407 광고 규제 케이스 3, CX L267~273 마케팅 케이스 어디에도 "본 자료는 투자 권유가 아니며 투자 결정은 본인 책임" 류 면책 표시가 의무화되어 있지 않다.
- 금융소비자보호법 제17조(적합성)·제18조(적정성)·제19조(설명의무)·제21조(부당권유 금지) 및 자본시장법 제49조(불건전 영업행위 금지)·제178조(부정거래)는 광고·안내 본문에 투자판단의 자기책임 원칙·과거 수익률 미래 보장 부인·원금손실 가능성을 명시하도록 한다. CX 마케팅 심의문구 3종은 협회 광고심의용 최소 표기일 뿐, **금소법 제19조의 "설명의무 표준 면책 문구"는 별도**다.
- **패치**: CX 마케팅·뉴스레터·이메일 전 케이스에 면책 의무 문구 셋(자기책임·과거수익률 미래 미보장·세제 변동 가능)을 가이드 Np 매핑 표에 추가하고, LLM이 누락 감지 시 ⚠️ 출력. 자본시장법 제178조 시세조종·부정거래 표현(예: "지금 사면 무조건 오릅니다", "내부정보", "선반영" 등)도 단정 표현 블랙리스트에 추가(현재 UX L195~198에는 "수익률 보장"만 등재).

### [HIGH-4] 모델 거버넌스 — 시스템 프롬프트 변경 이력·승인·롤백·감사 추적 부재
- HANDOFF L86~155 (CX v5.21~v6.0, 직원용 v5~v11.0)에서 버전 이력은 마크다운 표로만 존재하며, 다음이 모두 누락:
  (a) 변경 승인자 (준법감시·정보보호·CISO·CDO·CPO 서명),
  (b) 운영 환경 적용 시점·롤백 절차,
  (c) 변경 전후 회귀 평가셋·금융 컴플라이언스 평가(편향·차별·환각율) 결과,
  (d) 시스템 프롬프트 자체의 무결성(SHA-256·서명) 보호.
- 전자금융감독규정 제8조(전자금융기반시설 변경관리)·제13조(외부주문 변경관리)·제30조(전산자료 보호대책)·제37조의4(클라우드 정보처리 안전성 확보) 및 금융분야 AI 가이드라인(2021) "AI 모델 변경관리·책임자 지정·내부통제" 미충족.
- **패치**: 별도 거버넌스 문서(`06_운영_메타/거버넌스/`)에 (1) 시스템 프롬프트 변경 PR 템플릿(준법·정보보호 2인 승인), (2) 평가셋 회귀 게이트(50건+편향·환각·PII누설), (3) 롤백 SOP(v6.0 → v5.45 즉시 복귀 트리거), (4) 프롬프트 파일 해시 저장 의무 명시. CX/직원용/UX 첫 줄에 `version_id`, `approved_by`, `approved_date`, `sha256` 메타 헤더 추가.

### [MEDIUM-5] 환각·차별·편향 모니터링 룰 부재 — 금융분야 AI 가이드라인 위반
- CX L478~516, 직원용 L196~217, UX L167~198 모두 "메타 문구·가짜 원문·placeholder 훼손" 등 형식 환각만 다루며, **금융 AI 의사결정에서 더 중요한 의미 환각·차별·편향 모니터링은 전무**.
- 예: 직원용 EX3(L386~403)에서 `김*영 고객 계좌 110-12-345678` 같은 PII가 LLM 출력에 그대로 등장. 환각 또는 다른 고객 PII와 혼합될 경우 개인정보보호법 제29조(안전조치) 위반·신용정보법 제19조(기술적·물리적·관리적 보호조치) 위반 사고가 된다. 또한 포용 언어(가이드 137p) 룰만으로는 금융분야 AI 가이드라인의 "차별 금지(연령·성별·장애 등)" 의무를 충족하지 않는다.
- **패치**: (a) 출력 단계 PII 패턴 감지 후처리(주민번호·카드번호·계좌번호 정규식), (b) 차별·혐오 키워드 블랙리스트(노인·장애·성별 차등 표현), (c) 일별/주별 환각율·편향 모니터링 대시보드 및 임계치 초과 시 자동 알림. 시스템 프롬프트에는 `[차별·편향 차단] 연령·성별·지역·장애·국적 기반 차등 표현 발견 시 출력 거부` 룰 추가.

### [MEDIUM-6] 외부 위탁 LLM 사고 보고 의무·정보유출 대응 절차 부재
- HANDOFF L29 `모델: GPT-OSS-120B`. 모델 운영 주체가 외부인지 사내인지 불분명하나, 외부 호스팅이라면 전자금융감독규정 제13조·제14조의2(외부주문)·제43조(사고 보고) 및 신용정보법 제39조의4(개인신용정보 누설 통지·신고)·개인정보보호법 제34조(유출 통지·신고)에 따라 **사고 인지 후 24시간 내 신고** 의무.
- 시스템 프롬프트 어디에도 "정보유출 의심 패턴 감지 시 운영자 알림" 룰이 없다.
- **패치**: HANDOFF에 사고 대응 SOP 추가:
  (a) LLM 응답에 입력에 없는 PII가 등장하면 즉시 차단 + 운영자 통보,
  (b) 24시간 내 금감원·KISA 신고 책임자 지정,
  (c) 외주 LLM 계약서에 정보처리 목적 외 학습·재사용 금지 조항 명시(전자금융감독규정 제13조 제2항).

### [MEDIUM-7] 모델 출력 로그 보존 의무 미명시
- 어느 프롬프트에도 출력 로그 보존 정책이 없다. CX는 고객 응대 메시지를 생성하므로 「전자상거래 등에서의 소비자보호에 관한 법률」 제17조 및 시행령 제6조(거래 기록 보존 5년), 신용정보법 제20조의2(처리 기록 3년), 전자금융거래법 제22조(전자금융거래 기록 5년) 적용. **LLM 입출력은 마케팅·고지 콘텐츠 생성 도구로서 "기록"의 일부**다.
- **패치**: HANDOFF L424~437 운영 환경 섹션에 로그 보존 정책 추가: (a) 입력·출력 전체 텍스트 5년, (b) 시스템 프롬프트 SHA-256·버전 ID 함께 저장, (c) 무결성·접근통제(전자금융감독규정 제13조 제4항, 제30조), (d) 로그 안 PII 마스킹 또는 키 분리 저장.

### [LOW-8] 회사명·임직원·지점명 노출 정책 일관성 부재
- CX L772 verbatim `미래에셋증권 #{{계좌관리점명}} #{{계좌관리점대표번호}}` (placeholder, OK)
- 직원용 L168 `호칭 그대로 유지 (부장님·과장님·팀장님·선임·대표님·이사님)`, EX2(L367) `김 부장님` 실명 노출.
- 직원용은 사내 도메인이라도 LLM 외부 전송 시 임직원 실명·직책은 신용정보법 제32조 제6항(신용정보회사등 임직원 정보)·개인정보보호법 제15조에 따라 보호 대상.
- **패치**: 직원용에서도 임직원 실명 마스킹(`김 부장님 → ○ 부장님`) 또는 직책만 유지. 지점명·관리점명·VOC전담팀 명칭이 영업비밀에 해당할 수 있는 경우 placeholder 사용.

### [LOW-9] 정보 분류(대외비/내부 한정) 라벨링 룰 부재
- 직원용 어디에도 입력 메시지의 정보 분류(대외비·내부 한정·공개) 표시 의무가 없다. 사내 보안 규정에서 통상 요구되는 "C2/C3 등급 메시지는 외부 LLM 송출 금지" 게이트가 LLM 측에서 1차 차단되지 않는다.
- 전자금융감독규정 제13조의2(전자금융기반시설 보호 대책)·제30조(전산자료 보호대책)·금융분야 AI 가이드라인 "데이터 거버넌스" 권고 미흡.
- **패치**: 직원용 입력 첫 줄에 `[정보등급: 공개/내부/대외비/극비]` 태그 의무화. `대외비/극비` 감지 시 LLM은 처리 거부 + 사내 보안 도구 사용 안내.
