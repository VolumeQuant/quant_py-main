# 잠정실적(공정공시) 활용 연구 (2026-04-17)

> 중학생도 이해할 수 있게 쉽게 정리한 문서입니다.
> 집PC에서 pull 받아 이어 작업할 수 있도록 시행착오 전체를 상세히 기록합니다.

---

## 1. 잠정실적이 뭐야?

회사가 분기(3개월)마다 "우리 이번에 이만큼 벌었어요"라고 발표하는 게 **분기보고서**.
근데 분기보고서는 회계 감사를 받아야 해서 **분기 끝나고 45일** 정도 걸린다.

**잠정실적**은 감사 전에 "대충 이 정도입니다"라고 **먼저 공개**하는 것.
분기 끝나고 **2~4주** 만에 나온다. 정식보다 2~4주 빠른 "속보".

### 비유
- 시험 끝나고 **채점 전 예상 점수** = 잠정실적 (빠르지만 약간 다를 수 있음)
- **정식 성적표** = 분기보고서 (정확하지만 느림)

## 2. 왜 관심 가졌나?

우리 시스템의 **Growth(성장) 팩터**는 "매출이 얼마나 늘었나"를 본다.
현재는 정식 분기보고서(45일 후)가 나와야 업데이트됨.

**잠정실적을 쓰면 2~4주 먼저 "이 회사 매출 엄청 늘었네!" 파악 가능.**
→ 다른 사람보다 빨리 좋은 종목 잡을 수 있지 않을까?

## 3. 실제 데이터로 확인한 것

### 3.1 잠정실적에 뭐가 들어있나? (브이엠 2026.1Q 실제 예시)

```
구분(단위: 백만원)    당기(26.1Q)    전기(25.4Q)    전년동기(25.1Q)    전년동기 대비
매출액              88,893        50,845        17,908         +396%
영업이익             30,143         9,327         1,880       +1,504%
당기순이익            25,580        11,161         1,565       +1,534%
지배주주 순이익         25,580        11,161         1,565       +1,534%
```

**들어있는 것** (4개): 매출액, 영업이익, 당기순이익, 지배주주순이익
**없는 것**: 자산, 자본, 부채, 매출총이익, 영업현금흐름, 매출원가 등 (30개+)

### 3.2 분기당 몇 종목이 공시하나?

과거 4분기 전수 조사 (DART API 실측):

| 분기 | 공시 종목수 | 대형주(Y) | 코스닥(K) |
|---|---|---|---|
| 23.1Q | **232** | 159 | 103 |
| 23.2Q | **254** | 166 | 120 |
| 24.2Q | **238** | 158 | 120 |
| 25.2Q | **226** | 142 | 120 |
| 25.4Q (연간) | **43** | 27 | 22 |

**분기당 200~250종목이 공시** (연간 결산인 4Q만 적음).
처음에 "30종목밖에 안 돼"라고 했는데 **1주일치만 보고 판단한 실수**. 시즌 전체(2개월)로 보면 200+.

### 3.3 우리 유니버스(300종목)와 겹침

| 기준 | 겹침 |
|---|---|
| 우리 유니버스 전체 | **64종목 (24%)** |
| Top 20 | **3종목 (15%)** |

처음 "3.5%"라고 했다가 → 이름 매칭 오류 + 1주일 데이터만 봄 → 실제 **24%**.

### 3.4 파싱 가능한가?

**가능.** DART API로 목록 조회 → ZIP 다운로드 → HTML 파싱(`pd.read_html`).
테이블 구조가 표준화돼 있어 자동 추출 가능.

단, **구조화 API는 없다** (목록만 API, 수치는 HTML 파싱 필수).

## 4. 우리 Growth 팩터에 반영할 수 있나?

### 4.1 Growth 서브팩터 6개 vs 잠정실적

| 서브팩터 | 의미 | 필요한 데이터 | 잠정에 있나? |
|---|---|---|---|
| **rev_yoy** | 매출 성장률 | 매출액 | ✅ 있음 |
| **oca** | 영업이익 변화/자산 | 영업이익 + **자산** | ❌ 자산 없음 |
| **gp_yoy** | 매출총이익 성장 | **매출총이익** | ❌ 없음 |
| **op_margin_chg** | 이익률 변화 | 매출 + 영업이익 | ✅ 있음 |
| **cfo_yoy** | 현금흐름 성장 | **영업현금흐름** | ❌ 없음 |
| **rev_accel** | 매출 가속도 | 매출액 | ✅ 있음 |

**6개 중 3개만 가능.** oca(30%)와 gp(20%)는 불가 — 자산, 매출총이익 데이터 없음.

### 4.2 v79 공격 모드에 미치는 영향

공격 모드 Growth = rev_z(50%) + oca_z(30%) + gp_growth_z(20%)
잠정으로 선행 가능: **rev_z(50%)만**. 나머지 50%(oca+gp)는 이전 분기 유지.

→ 전체 점수의 50% × G비중 50% = **25%만 2~4주 선행**. 나머지 75%는 그대로.

### 4.3 oca 대안: 직전 분기 자산 사용

oca = (영업이익 변화) / **자산**
잠정에 자산 없으면 → **직전 분기 자산 그대로 사용 가능**.
자산은 분기 간 변동이 작아(±5%) 오차 미미.
현재 코드도 이미 "자산 없으면 이전값 사용" 구조라 추가 수정 불필요.

## 5. 전문가 컨설팅 결과 (퀀트 PM 10년 경력)

### 5.1 핵심 결론: "기존 시스템에 통합하지 마라"

**이유 3가지**:

**① 커버리지 편향 (Selection Bias)**
잠정실적 빨리 내는 회사 = 실적 좋은 회사. "좋은 뉴스만 빨리 공개".
→ 잠정 반영하면 "좋은 회사만 빨리 점수 올라감" → BT 성과 과대평가.
→ 실전에서는 "잠정 안 낸 회사 중에서도 좋은 회사 있는데 못 잡음".
→ **BT에서 +0.3 개선돼도 실전 기대치는 +0.1~0.2 수준** (30~50% 편향).

**② 파이프라인 복잡도**
현재 이미 DART + FnGuide + pykrx 3소스 관리. 여기에 잠정실적 추가하면:
- 잠정/확정 이중 반영 로직
- HTML 파싱 유지보수
- 잠정→확정 교체 시 ranking 급변 가능
→ **버그 유입 리스크** (v76~v78 마이그레이션 시행착오 반복).

**③ 효용 부족**
공격 모드에서 전체 점수의 25%만 선행. weighted_rank 3일 가중 시스템에서 "하나의 서브팩터가 2~4주 빨라지는 것"이 유의미한 랭킹 변동을 일으키기 어려움.

### 5.2 전문가 유일 추천: "텔레그램 알림만"

Growth 팩터에 통합하지 말고, "오늘 잠정실적 공시 종목 + 영업이익 YoY" 알림만 가볍게 추가.
파이프라인 변경 없음, 개발 1일, 유지보수 0.

## 6. 우리의 새 아이디어: 버프/너프 시스템

전문가 의견 "기존 시스템 건드리지 마라"를 수용하되, **점수에 보너스/페널티**를 주는 방식:

### 6.1 기본 컨셉

기존 멀티팩터 점수(V+Q+G+M) 계산은 **그대로 유지**.
잠정실적 공시된 종목에 대해 **별도 보너스/페널티 점수**를 weighted_rank에 반영.

```
최종_wr = 기존_wr + 잠정실적_버프너프
```

### 6.2 버프/너프 아이디어 (검토 필요)

**버프 (점수 보너스, wr 감소 = 순위 상승)**:
- 잠정 영업이익 YoY > +50% → wr -1.0 (1단계 상승 효과)
- 잠정 영업이익 YoY > +100% → wr -2.0
- 잠정 매출 YoY > +30% → wr -0.5

**너프 (점수 페널티, wr 증가 = 순위 하락)**:
- 잠정 영업이익 YoY < -30% → wr +1.0
- 잠정 영업이익 적자전환 → wr +2.0
- 잠정 매출 YoY < -20% → wr +0.5

**버프/너프 지속 기간**: 잠정 공시일 ~ 정식 보고서 공시일 (2~4주)
정식 보고서 나오면 자동 해제 (기존 Growth 팩터가 업데이트되니).

### 6.3 장점
- 기존 멀티팩터 계산 로직 **안 건드림** (ranking JSON 변경 없음)
- weighted_rank 후처리 단계에서 보너스만 적용 → 구현 간단
- BT에서 버프/너프 크기 최적화 가능 (그리드서치)
- 잠정 안 낸 종목은 영향 없음 (기존 그대로)

### 6.4 주의사항
- Selection bias 여전히 존재 (잠정 내는 종목만 버프 받으니)
- 버프 크기가 너무 크면 잠정 공시 여부 자체가 팩터가 됨
- BT 시 잠정실적 데이터 전수 수집 필요 (2023~2026, ~2,400건)
- 잠정 vs 확정 괴리 발생 시 버프→너프 급반전 가능

### 6.5 구현 계획 (미확정, 집PC에서 이어갈 것)

```
Step 1: 잠정실적 전수 수집 (2023~2026, DART API + HTML 파싱, ~80분)
Step 2: 잠정 vs 확정 괴리율 EDA (얼마나 다른지)
Step 3: 괴리 미미 → 버프/너프 크기 그리드서치 (BT)
Step 4: 표본 테스트 (3개월) → 전체 BT
Step 5: 성과 비교 (버프 있음 vs 없음)
```

## 7. 시행착오 기록 (반복 방지)

### 7.1 "30종목밖에 안 돼" 오판
- 1주일치 데이터(4/10~17)만 보고 "분기당 30종목"이라 결론
- 실제 2개월 시즌 전체: **200~250종목**
- **교훈**: 기간 샘플이 짧으면 과소추정. 반드시 전체 시즌 확인.

### 7.2 "중소형주는 안 한다" 오판
- 에이전트 조사에서 "중소형주 절반 이상 미제출" 보고
- 실제 데이터: **코스닥(K) 비율 40~50%** — 중소형주도 활발히 공시
- **교훈**: 에이전트 조사 결과를 검증 없이 수용하면 안 됨. 직접 API 호출해서 확인.

### 7.3 이름 매칭 실패 → ticker 매칭
- 처음 종목명 문자열 매칭 → "SK하이닉스" vs "에스케이하이닉스" 불일치
- DART API의 `stock_code` 필드로 ticker 매칭 → 정확도 향상
- **교훈**: 종목 매칭은 항상 ticker(종목코드) 기반.

### 7.4 Growth 선행 효과 과대평가
- 처음 "6개 서브팩터 전부 선행 가능"이라 기대
- 실제: **3개만 가능** (자산, 매출총이익, 현금흐름 없음)
- 공격 모드에서 전체 25%만 선행 → 효과 제한적
- **교훈**: 데이터 스키마 먼저 확인 후 효용 판단.
